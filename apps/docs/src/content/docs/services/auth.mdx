Identity gives a NubleStation org **cross-platform user accounts** ‚Äî one identity per human, usable across every app ‚Äî with **per-app authorization**. A user registers once at `identity.{org}.local`, and a session cookie scoped to the parent domain logs them in everywhere. App access is **default-deny**: a Console admin grants a role per app.

<Aside type="tip" title="Shipped">
  Identity is live. For the client library, see [`@nublestation/identity`](/docs/sdk/auth).
</Aside>

## The model in one line

> Register once at `identity.{org}.local`; a session cookie scoped to `.{org}.local` signs you in across every app; an admin grants you a role per app (default-deny).

There are **no browser-side tokens**. Authentication is a server-side, revocable session referenced by an `HttpOnly` cookie. This is shared-cookie SSO, chosen over full OIDC because every app is first-party under one parent domain ‚Äî it delivers SSO while eliminating OAuth's browser-token attack surface (open redirects, code interception, `localStorage` token theft).

## The two reach paths

```
identity.{org}.local/*       ‚Üí  Caddy  ‚Üí  identity:3004                 (pages + form posts; sets the cookie)
api.{org}.local/v1/auth/*    ‚Üí  Caddy  ‚Üí  gateway  ‚Üí  identity:3004     (JSON; cookie passed through)
```

The Gateway forwards `/v1/auth/*` **without** requiring an API key ‚Äî it passes the `Cookie` header through and relays `Set-Cookie` back.

```
tasks.{org}.local  ‚îÄ‚îÄ"Login"‚îÄ‚îÄ‚ñ∂  identity.{org}.local/authorize?app=tasks&redirect_uri=http://tasks.{org}.local/

                        ‚îî‚îÄ yes ‚ñº

                        ‚îú‚îÄ has role ‚ñ∂ 302 redirect_uri   (validated: host ends with .{org}.local)
                        ‚îî‚îÄ none     ‚ñ∂ "No access" page (default-deny)
```

`redirect_uri` is **allow-listed**: its host must be `{org}.local` or a `*.{org}.local` subdomain, blocking open redirects.

## Routes

### User-facing pages (direct via `identity.{org}.local`)

| Method | Path | Description |
|---|---|---|
| `GET`  | `/login` | Sign-in page (carries `?app` & `?redirect_uri` through the flow) |
| `POST` | `/login` | Verify credentials, create session, set cookie, continue |
| `GET`  | `/register` | Registration page (full name, email, password, optional avatar) |
| `POST` | `/register` | Create account (default-deny), upload avatar, sign in |
| `POST` | `/logout` | Revoke session, clear cookie, back to `/login` |
| `GET`  | `/authorize` | SSO grant flow (the login-button target) |
| `GET`  | `/account` | Signed-in landing (avatar, name, sign out) |

### JSON API (via Gateway `api.{org}.local/v1/auth/*`)

| Method | Path | Status | Description |
|---|---|---|---|
| `GET`  | `/v1/auth/me?app={slug}` | 200, 401, 403, 404 | Current user; with `app`, resolves role (403 if none) |
| `POST` | `/v1/auth/logout` | 200 | Revoke session, clear cookie |

## Sessions (revocable, server-side)

Backed by `platform.sessions` (`id`, `user_id`, `token_hash`, `expires_at`, `created_at`).

- The cookie carries a **32-byte CSPRNG token**; the DB stores only its **sha256** (`token_hash`). A DB read cannot reconstruct a usable cookie.
- **Revocable** ‚Äî logout deletes the row; an admin can force-log-out a user everywhere by deleting their rows.
- **Token rotation on login** ‚Äî every login mints a fresh token, so a pre-set (fixated) token is never elevated.
- TTL: `SESSION_TTL_HOURS` (default 8).

### Cookie attributes (HTTP-only deployment)

| Attribute | Value | Why |
|---|---|---|
| `HttpOnly` | on | Blocks JS/XSS theft (works over HTTP) |
| `SameSite` | `Lax` | CSRF backstop; `Lax` so the authorize redirect carries the cookie |
| `Domain` | `.{org}.local` | Sent to every app subdomain ‚Äî the SSO backbone |
| `Secure` | **off for now** | Requires HTTPS; flip via `SECURE_COOKIES=true` once TLS lands |

<Aside type="caution" title="HTTP-era limitation">
  Without TLS, a same-LAN sniffer could capture the session cookie. This is the one accepted trade-off; it's resolved the moment Caddy HTTPS (internal CA) is enabled ‚Äî only `SECURE_COOKIES` changes. Every other control stays at maximum.
</Aside>

## Authorization

`getUserAppRole(userId, appId)`:

1. If the user's `platform.users.role` is `super_admin` or `admin` ‚Üí **`admin`** (implicit on every app; no `user_app_access` row needed).
2. Otherwise ‚Üí their `platform.user_app_access` role for that app, or **`null`** (default-deny).

Admins are the **grantors**. Console ‚Üí App ‚Üí **Users** tab grants, revokes, and changes end-user roles (writing `platform.audit_log`). A freshly registered user has **zero** app access until an admin acts.

## Avatars

Profile photos live in a **reserved system Vault bucket** (app slug `identity-system`, seeded idempotently at boot). On registration Identity HMAC-signs an upload to [Vault](/docs/services/storage) as the system app, makes the file public, and stores the URL in `platform.users.avatar_url`. Avatars are therefore cross-app, not scoped to the app a user registered from.

```
http://api.{org}.local/vault/identity-system/avatars/{userId}.{ext}
```
