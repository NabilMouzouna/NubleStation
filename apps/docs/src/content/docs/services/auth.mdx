---
title: "Identity — Auth Service"
description: Cross-app user accounts with shared-cookie single sign-on and per-app, default-deny authorization.
---

Identity gives a NubleStation org **cross-platform user accounts** — one identity per human, usable across every app — with **per-app authorization**. A user registers once at `identity.{org}.local`, and a session cookie scoped to the parent domain logs them in everywhere. App access is **default-deny**: a Console admin grants a role per app.

<Aside type="tip" title="Shipped">
  Identity is live. For the client library, see [`@nublestation/identity`](/docs/sdk/auth).
</Aside>

## The model in one line

> Register once at `identity.{org}.local`; a session cookie scoped to `.{org}.local` signs you in across every app; an admin grants you a role per app (default-deny).

There are **no browser-side tokens**. Authentication is a server-side, revocable session referenced by an `HttpOnly` cookie. This is shared-cookie SSO, chosen over full OIDC because every app is first-party under one parent domain — it delivers SSO while eliminating OAuth's browser-token attack surface (open redirects, code interception, `localStorage` token theft).

## The two reach paths

```
identity.{org}.local/*       →  Caddy  →  identity:3004                 (pages + form posts; sets the cookie)
api.{org}.local/v1/auth/*    →  Caddy  →  gateway  →  identity:3004     (JSON; cookie passed through)
```

The Gateway forwards `/v1/auth/*` **without** requiring an API key — it passes the `Cookie` header through and relays `Set-Cookie` back.

## The SSO "login button" flow

```
tasks.{org}.local  ──"Login"──▶  identity.{org}.local/authorize?app=tasks&redirect_uri=http://tasks.{org}.local/
                                       │
                       session cookie on .{org}.local?
                        ├─ no  ▶ render login/register ▶ on success ▼
                        └─ yes ▼
                       role for (user, tasks)?
                        │   admin if user is super_admin/admin, else user_app_access lookup
                        ├─ has role ▶ 302 redirect_uri   (validated: host ends with .{org}.local)
                        └─ none     ▶ "No access" page (default-deny)
                                       │
tasks.{org}.local  ◀──redirect──  app calls GET api.{org}.local/v1/auth/me?app=tasks
                                   ▶ { id, email, displayName, avatarUrl, role }   (403 if no access)
```

`redirect_uri` is **allow-listed**: its host must be `{org}.local` or a `*.{org}.local` subdomain, blocking open redirects.

## Routes

### User-facing pages (direct via `identity.{org}.local`)

| Method | Path | Description |
|---|---|---|
| `GET`  | `/login` | Sign-in page (carries `?app` & `?redirect_uri` through the flow) |
| `POST` | `/login` | Verify credentials, create session, set cookie, continue |
| `GET`  | `/register` | Registration page (full name, email, password, optional avatar) |
| `POST` | `/register` | Create account (default-deny), upload avatar, sign in |
| `POST` | `/logout` | Revoke session, clear cookie, back to `/login` |
| `GET`  | `/authorize` | SSO grant flow (the login-button target) |
| `GET`  | `/account` | Signed-in landing (avatar, name, sign out) |

### JSON API (via Gateway `api.{org}.local/v1/auth/*`)

| Method | Path | Status | Description |
|---|---|---|---|
| `GET`  | `/v1/auth/me?app={slug}` | 200, 401, 403, 404 | Current user; with `app`, resolves role (403 if none) |
| `POST` | `/v1/auth/logout` | 200 | Revoke session, clear cookie |

## Sessions (revocable, server-side)

Backed by `platform.sessions` (`id`, `user_id`, `token_hash`, `expires_at`, `created_at`).

- The cookie carries a **32-byte CSPRNG token**; the DB stores only its **sha256** (`token_hash`). A DB read cannot reconstruct a usable cookie.
- **Revocable** — logout deletes the row; an admin can force-log-out a user everywhere by deleting their rows.
- **Token rotation on login** — every login mints a fresh token, so a pre-set (fixated) token is never elevated.
- TTL: `SESSION_TTL_HOURS` (default 8).

### Cookie attributes (HTTP-only deployment)

| Attribute | Value | Why |
|---|---|---|
| `HttpOnly` | on | Blocks JS/XSS theft (works over HTTP) |
| `SameSite` | `Lax` | CSRF backstop; `Lax` so the authorize redirect carries the cookie |
| `Domain` | `.{org}.local` | Sent to every app subdomain — the SSO backbone |
| `Secure` | **off for now** | Requires HTTPS; flip via `SECURE_COOKIES=true` once TLS lands |

<Aside type="caution" title="HTTP-era limitation">
  Without TLS, a same-LAN sniffer could capture the session cookie. This is the one accepted trade-off; it's resolved the moment Caddy HTTPS (internal CA) is enabled — only `SECURE_COOKIES` changes. Every other control stays at maximum.
</Aside>

## Authorization

`getUserAppRole(userId, appId)`:

1. If the user's `platform.users.role` is `super_admin` or `admin` → **`admin`** (implicit on every app; no `user_app_access` row needed).
2. Otherwise → their `platform.user_app_access` role for that app, or **`null`** (default-deny).

Admins are the **grantors**. Console → App → **Users** tab grants, revokes, and changes end-user roles (writing `platform.audit_log`). A freshly registered user has **zero** app access until an admin acts.

## Avatars

Profile photos live in a **reserved system Vault bucket** (app slug `identity-system`, seeded idempotently at boot). On registration Identity HMAC-signs an upload to [Vault](/docs/services/storage) as the system app, makes the file public, and stores the URL in `platform.users.avatar_url`. Avatars are therefore cross-app, not scoped to the app a user registered from.

```
http://api.{org}.local/vault/identity-system/avatars/{userId}.{ext}
```

## Client SDK — `@nublestation/identity`

`@nublestation/identity` is the browser-side auth client. It answers "who is signed in?" and "do they have access to this app?", and drives the sign-in / sign-out redirect flow.

### Install

```bash
npm install @nublestation/identity
# or
pnpm add @nublestation/identity
```

### Create a client

```typescript
import { createIdentityClient } from "@nublestation/identity";

const auth = createIdentityClient({
  url:         "http://api.clinic.local",       // Gateway — reaches /v1/auth/*
  identityUrl: "http://identity.clinic.local",  // Identity pages — /authorize, /login
  app:         "tasks",                          // this app's slug from the Console
});
```

### Check the session

```typescript
const session = await auth.getSession();

switch (session.status) {
  case "authenticated":
    console.log(`Signed in as ${session.user.email} with role ${session.user.role}`);
    break;
  case "forbidden":
    // Signed in but no role on this app — show "ask an admin"
    break;
  case "unauthenticated":
    auth.login(); // redirect to identity.clinic.local/authorize
    break;
}
```

### Sign in / sign out

```typescript
// Redirect to the SSO login page, returning to the current URL after
auth.login();

// Revoke session server-side then navigate to the identity login page
await auth.logout();
```

### React: the `useIdentity` hook

The SDK is framework-agnostic. For React, copy the canonical hook and gate into your app (they are not bundled to keep the package dependency-free):

```typescript
import { useCallback, useEffect, useState } from "react";
import { createIdentityClient } from "@nublestation/identity";

export const identity = createIdentityClient({
  url:         import.meta.env.VITE_NUBLESTATION_URL,
  identityUrl: import.meta.env.VITE_NUBLESTATION_IDENTITY_URL,
  app:         import.meta.env.VITE_NUBLESTATION_APP,
});

export function useIdentity() {
  const [status, setStatus] = useState("loading");
  const [user,   setUser]   = useState(null);

  useEffect(() => {
    let cancelled = false;
    identity.getSession().then((s) => {
      if (cancelled) return;
      setStatus(s.status);
      setUser(s.status !== "unauthenticated" ? s.user : null);
    }).catch(() => { if (!cancelled) setStatus("unauthenticated"); });
    return () => { cancelled = true; };
  }, []);

  return {
    status,
    user,
    login:  useCallback(() => identity.login(), []),
    logout: useCallback(() => void identity.logout(), []),
  };
}
```

<Aside type="caution" title="Requires a deployed origin">
  Auth only works when your app runs on `{app}.{org}.local` — the cookie is scoped to `.{org}.local` and is never sent to `localhost`. Develop UI locally with mocked state; test the real SSO flow on the deployed app.
</Aside>
