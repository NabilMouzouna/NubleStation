`@nublestation/client` is the umbrella SDK: one factory, one config object, and a single client that exposes every NubleStation service your app uses. If you'd rather pull in just one service, the per-service packages ([`@nublestation/vault`](/sdk/storage/), [`@nublestation/identity`](/sdk/auth/)) work standalone — the unified client simply wraps them.

## Installation

```bash
npm install @nublestation/client
# or
pnpm add @nublestation/client
```

## Creating a client

```typescript
import { createClient } from "@nublestation/client";

const nuble = createClient({
  url:    "http://api.clinic.local",   // NubleStation gateway URL
  apiKey: "nbl_<key_id>.<secret>",     // API key from the Console
});
```

| Field | Description |
|---|---|
| `url` | Your gateway base URL, `api.{org}.local` |
| `apiKey` | App-scoped API key from **Console → Apps → your app → Settings** |

## Services

You call each service as `nuble.<service>.*`:

| Service | Accessor | Status |
|---|---|---|
| Vault — file storage | `nuble.vault` | **Live** |
| Blaze — database | `nuble.blaze` | Coming soon |

```typescript
// Upload, list, and download files through Vault
await nuble.vault.upload("reports", "q1.pdf", file);
const files = await nuble.vault.list("reports");
const bytes = await nuble.vault.download("reports", "q1.pdf");
```

The full Vault method set (`upload`, `download`, `list`, `listMine`, `listSharedWithMe`, `listPublic`, `setPublic`, `delete`, `share`, `unshare`, `listGrants`) is documented on the [`@nublestation/vault` page](/sdk/storage/) — every method is identical when reached through `nuble.vault`.

## Auth is configured separately

Authentication doesn't use an API key — it rides the organization's shared SSO session cookie, so it takes a different config (it needs your app slug and the Identity URL). For that reason auth lives in its own package, [`@nublestation/identity`](/sdk/auth/):

```typescript
import { createIdentityClient } from "@nublestation/identity";

const identity = createIdentityClient({
  url:         "http://api.clinic.local",
  identityUrl: "http://identity.clinic.local",
  app:         "bucket",
});

const session = await identity.getSession();
```

A typical app uses both side by side — `identity` to know **who** the user is, and `nuble.vault` to read and write **their** files.

## Error handling

Service calls throw a typed error on any non-2xx response. The Vault error class is re-exported from the client for convenience:

```typescript
import { VaultError } from "@nublestation/client";

try {
  await nuble.vault.upload("docs", "report.pdf", bytes);
} catch (err) {
  if (err instanceof VaultError) {
    console.error(err.status, err.code); // e.g. 401 "unauthorized"
  }
}
```

| Code | Status | Meaning |
|---|---|---|
| `unauthorized` | 401 | API key missing, invalid, or revoked |
| `forbidden` | 403 | No access to the resource |
| `not_found` | 404 | Resource does not exist |
| `internal_error` | 500 | Unexpected server error |

## Exports

```typescript
import { createClient, VaultError } from "@nublestation/client";
import type { ClientConfig, NubleClient, FileResult } from "@nublestation/client";
```

## When to use which package

- **`@nublestation/client`** — you use (or plan to use) more than one service and want a single dependency and config.
- **`@nublestation/vault` / `@nublestation/identity`** — you need exactly one service and want the smallest install.

Both approaches share the same method signatures and types, so moving between them is a one-line change.
