Role

The API Gateway is the only service published on your LAN. Every request from the SDK, the CLI, and app frontends goes through it. Internal services (DB, Auth, Storage, Deploy) listen only on the internal Docker network — they are unreachable from any LAN device without going through the gateway.

This single trust boundary means:

  • API key validation happens in exactly one place
  • Session validation happens in exactly one place
  • A compromised app container cannot impersonate another tenant

Authentication flow

API key requests (SDK / developer tools)

Authorization: Bearer nbl_<key_id>.<secret>
  1. Parse the header — split on . to get key_id and secret
  2. Look up key_id in Redis (cache hit → sub-millisecond). On miss, fall back to platform.api_keys in Postgres
  3. Check revoked_at and expires_at — reject if either applies
  4. Argon2id-verify secret against secret_hash — constant-time, rejects timing attacks
  5. Extract app_id from the resolved row

All four failure cases (not found, revoked, expired, wrong secret) return a generic 401 — no enumeration leakage.

Session requests (browser / Console)

The Console and app frontends authenticate via a session cookie scoped to Domain=.{org}.local. The gateway validates this cookie against the Auth service and resolves it to a user_id.

ℹ Note

Session-based auth is implemented in the Auth Service (currently in development). In Phase 1, X-Nuble-User-Id is set to the API key’s row UUID as a placeholder. Full user session resolution comes with the Auth service.

Request forwarding (HMAC signing)

After authentication, the gateway forwards requests to the appropriate internal service with a set of signed headers:

HeaderValue
x-nuble-app-idUUID of the authenticated app
x-nuble-user-idUUID of the authenticated user
x-nuble-timestampCurrent Unix timestamp (ms)
x-nuble-sigHMAC-SHA256 of METHOD\nPATH\nBODY_SHA256\nTIMESTAMP

The internal services verify this signature before trusting any header. If a compromised app container forges a request to the DB service, it cannot produce a valid signature without the INTERNAL_HMAC_SECRET env var — which only the gateway and the target service know.

See HMAC Request Signing for the full cryptographic detail.

Routing

The gateway routes by URL path prefix:

Path prefixForwarded to
/v1/auth/*Auth Service
/v1/db/*DB Service
/v1/storage/*Storage Service
/v1/deploy/*Deploy Service
/v1/admin/*Admin control API (admin session only)

Redis caching

API key resolution is on the hot path for every request. To avoid a Postgres round-trip per call, the gateway caches key_id → app_id in Redis with a short TTL.

If Redis is unavailable, the gateway falls back to Postgres — slower but correct. The cache is a performance optimization, not a correctness dependency.

Rate limiting and quotas

Per-app rate limits and request quotas are enforced at the gateway level (coming in Phase 6). Each app has a configurable requests-per-minute cap. Exceeding it returns 429 Too Many Requests.

Health endpoints

GET /healthz   → 200 OK (liveness — is the process running?)
GET /readyz    → 200 OK (readiness — can it serve traffic?)

Docker Compose uses these to determine when the gateway is ready to accept traffic after a restart.

Environment variables

VariableDescription
INTERNAL_HMAC_SECRETShared secret for signing forwarded requests. Must match across gateway + all services.
DATABASE_URLPostgres connection string (fallback for API key resolution when Redis is down)
REDIS_URLRedis connection string for API key cache
DB_SERVICE_URLInternal URL of the DB service (Docker network)
AUTH_SERVICE_URLInternal URL of the Auth service
STORAGE_SERVICE_URLInternal URL of the Storage service
DEPLOY_SERVICE_URLInternal URL of the Deploy service

All secrets are injected via infra/.env, generated by the installer.
