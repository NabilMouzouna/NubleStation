Current status

Section titled “Current status”

NubleStation is in active development. The infrastructure shell (Caddy, CoreDNS, Docker Compose, install script) is complete. The database service and API gateway are the current build focus.

In progress

Section titled “In progress”

Phase 1 — Foundation (current)

Section titled “Phase 1 — Foundation (current)”

Goal: prove cross-tenant isolation end-to-end.

  • Platform schema (organizations, users, apps, api_keys, user_app_access, migrations, audit_log)
  • Connection manager with SET LOCAL app.current_tenant pattern
  • HMAC signing middleware (gateway + DB service)
  • Cross-tenant isolation test green in CI

Phase 2 — Built-in users resource

Section titled “Phase 2 — Built-in users resource”
  • platform.users CRUD via internal admin API
  • tenant_data.users view filtered by user_app_access
  • SDK methods: nuble.users.findBy(), nuble.users.create()
  • Isolation test: App A cannot see App B’s users through the view

Phase 3 — Schema DSL

Section titled “Phase 3 — Schema DSL”
  • packages/schema workspace
  • DSL parser (Zod) → validated structure
  • Generator: DSL → SQL, auto-inject app_id and RLS policy
  • Migration runner library (callable from CLI and API)
  • Migration safety: pg-query-parser, checksum verification, advisory lock
  • REST router + query validator + query builder
  • Isolation test: a developer-defined table works end-to-end with RLS

Phase 4 — SDK packaging

Section titled “Phase 4 — SDK packaging”
  • ADR 004: SDK packaging strategy (multi-entry, tree-shakeable)
  • Type generation from DSL → .nuble/types.ts
  • Builder chain: .where(), .orderBy(), .include(), .limit()
  • Bundle check: a /db-only app contains no auth/storage code

Phase 5 — Real-time + escape hatches

Section titled “Phase 5 — Real-time + escape hatches”
  • Auto-generated triggers → NOTIFY on tenant table changes
  • SSE endpoint + SDK .subscribe()
  • Level 1: computed fields (generated columns)
  • Level 2: named queries (reviewed, logged)

Phase 6 — Hardening

Section titled “Phase 6 — Hardening”
  • Per-request quotas + statement_timeout
  • Audit logging for all sensitive operations
  • PgBouncer transaction pooling validation under load
  • Backup script: pg_dump + filesystem snapshot

Out of scope (permanent)

Section titled “Out of scope (permanent)”

These will not be added to NubleStation:

  • Multi-machine clustering (Kubernetes, Docker Swarm)
  • Internet-facing hosting (NubleStation is LAN-native by design)
  • Server-side rendering for deployed apps (static SPAs only)
  • Built-in CI/CD pipelines for deployed apps
  • S3-compatible storage API (local filesystem only)
  • Mobile native SDKs
  • Local LLM inference API
  • WebSocket real-time in v1 (SSE covers the use cases; WebSockets are v2)

Defense day milestone

Section titled “Defense day milestone”

The project defense targets a live demo flow:

  1. Fresh Ubuntu VM → curl ... | bash → console live in under 10 minutes
  2. Admin creates an app “tasks” → reserves tasks.clinic.local
  3. Developer deploys a task management SPA → visible on any LAN device
  4. Unplug internet cable → everything still works
  5. Kill a container → Docker auto-restarts it
  6. RLS demonstrated: App A cannot read App B’s data

Version 1.0.0 is tagged on defense day.
